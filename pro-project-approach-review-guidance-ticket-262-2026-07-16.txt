Recommendation: KEEP, but adjust

Keep final reconciliation as the authoritative fix point. Do not pivot to a schedule-emission-only fix.

The current diagnosis is consistent with main at v2.12.867: free-import classification initially forces charge, while both schedule emission and final reconciliation unconditionally turn a charge clipped to at most 100 W into self_consumption. Production applies reconciliation after spread/bridge/overlay processing, and neither reconciliation call currently receives the effective No Idle setting.

The candidate patch has the correct overall shape, but it needs two important adjustments:

Choose the replacement action from preserved model-flow evidence, not merely from the fact that charge was clipped.

Make that decision before SOC is advanced, and remove the clipped grid-charge component from the emitted battery flow.

Otherwise, a label-only change can produce an idle action with nonzero charge or an SOC increase that never physically occurred.

Schedule emission versus final reconciliation
Schedule-emission fix

Emission has two advantages:

It already receives disable_idle.

It has direct access to the LP or greedy flow vectors when mapping them to actions. The ordinary hold classification already treats near-zero battery movement plus grid import as a candidate for idle, with No Idle converting ordinary holds to self_consumption.

However, an emission-only fix is incomplete:

The free-import override deliberately manufactures a charge action even where the model selected zero battery charge.

Spread-import and other post-processing can change the schedule after emission.

Final reconciliation replays SOC and clips charge again against physical 100% and the configured grid-charge cap.

Publication, costing, graph flows, and execution use the reconciled result.

Therefore, correcting only _build_schedule still leaves the same defect available in reconcile_result_with_schedule.

Final-reconciliation fix

Final reconciliation is the right mandatory seam because it sees:

The actual schedule that will be published.

The cumulative SOC after all earlier slots.

Physical charge headroom.

Grid-charge-cap headroom.

Post-processing changes.

It is also where final grid flows and economics are recalculated from the emitted schedule.

The safest scoped approach is:

Put the policy in reconciliation.

Do not duplicate a second, subtly different policy in emission.

Extract a private helper only if both call sites genuinely need identical behavior or direct optimize() consumers must receive the corrected intermediate action.

For the reported production path, reconciliation alone is sufficient and lower risk.

Safest IDLE predicate

Do not use any of these as the deciding signal:

Free import price by itself.

Presence of a later high export price.

SOC equality with the cap.

effective_grid_charge_w <= 100 by itself.

The post-processed action label alone.

Those signals do not prove that the optimizer intended to preserve SOC by importing the home load.

There is no natural solar charge that an IDLE action would incorrectly suppress.

The equality should be tolerance-based, not a strict float comparison. Using the existing 100 W action threshold is consistent with the rest of the action mapper.

No Idle behavior

Preserve the effective No Idle behavior exactly:

Cross-brand risks
IDLE is not implemented identically across brands

The optimizer’s conceptual IDLE means “hold SOC while the grid supplies load,” but active execution is heterogeneous:

Some systems use a backup/hold mode and an elevated reserve.

GoodWe intentionally falls back to self-consumption for ordinary IDLE.

Some generic backends may only restore normal mode and cannot guarantee a hard hold.

Existing generic-brand drift detection is limited.

Do not add battery-brand checks to the optimizer predicate. The published recommendation should represent the modeled energy trajectory. Brand-specific inability to execute an existing IDLE action is a separate controller defect and needs its own reproduction.

SAJ and solar

The SAJ controller exposes a passive zero-power idle operation that holds the battery, but its documented behavior also prevents PV charging while idle. This makes the “no meaningful natural solar charge” guard essential. A broad “clipped charge means IDLE” rule could export solar that the optimizer expected the battery to absorb.

That controller capability does not prove what happened for the reporter: Monitoring Mode prevented execution entirely.

Increased mode transitions

This correction may introduce additional charge → idle → export transitions in active mode. Existing restore and reserve-state behavior must therefore remain untouched, and the existing controller test suites should be run even though the patch is schedule-only.

SOC jitter

Do not classify based on soc == cap or soc == 1.0. Battery readings and interval calculations can jitter. Let physical clipping and the existing 100 W action threshold determine whether force charge remains meaningful.

Runtime demand-window override

Runtime can already replace an IDLE action with self-consumption inside a demand window. Do not alter that policy as part of this defect.

Relabelling after SOC advancement. This can publish an IDLE action with an SOC increase from charge that was removed.

Checking only grid charge. A slot with no grid charge but substantial solar charge must be self_consumption, not idle.

Using the raw No Idle option. Pass the effective provider-gated value captured for the solve.

Leaving a residual grid-charge flow on self_consumption. Strip the grid component whenever the force-charge action is discarded.

Expanding into controller changes. The demonstrated bug is the published schedule classification; brand execution changes need separate evidence.
