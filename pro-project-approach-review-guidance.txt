PowerSync iOS dashboard scroll review (2026-07-15)

First review response:

## Recommendation: **adjust**

Do **not** release the host-level monotonic max-height ratchet as the sole fix. Make the **per-item transient height lock** the primary mechanism, and retain a host-height floor only as a narrowly scoped backstop while validating the new implementation on iOS. Defer the atomic/static-shell card refactor until after this release.

The current ratchet is valuable evidence and a good diagnostic: it directly prevents the document’s scroll range from temporarily becoming smaller than the current offset, and the reporter has already verified that mitigation on-device. But it is too broad and too persistent to be the final architecture. It can preserve the total scroll range while still allowing cards below a collapsing item to move, and its correctness depends on identifying every operation that can legitimately reduce dashboard height.

The browser model matches the reported failure: CSSOM View constrains a vertical scroll position to the scrolling-area height minus the viewport height. Separately, scroll anchoring is intended to compensate for layout changes above the viewport. WebKit describes scroll anchoring as new in Safari 27 beta and also lists an iOS fix for programmatic scrolling interrupting momentum, so neither `overflow-anchor` nor manual `scrollTo` compensation is an appropriate primary solution for deployed HA WKWebViews. ([W3C][1])

## Approach comparison

| Approach                              | What it reliably protects                                                                                                          | Main failure mode                                                                                                              | Recommended role                                      |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| Monotonic host max-height             | Prevents total scroll range from temporarily shrinking below its previous maximum                                                  | Stale blank tail; incomplete resets; does not necessarily preserve positions inside the host                                   | Diagnostic and temporary backstop, not sole final fix |
| Per-item transient floors             | Prevents each card wrapper from exposing an intermediate collapsed height, preserving both total range and downstream item offsets | Unlock timing, stale generations, and measuring the locked wrapper rather than the natural card                                | Primary release fix                                   |
| Off-DOM atomic swap                   | Can reduce or eliminate an individual card’s teardown interval                                                                     | Detached charts may measure at zero size; custom-element initialization can still be asynchronous; only fixes refactored cards | Not the release path                                  |
| Static shell with incremental updates | Eliminates teardown for known cards and improves rendering efficiency                                                              | Larger behavioral refactor and wider regression surface                                                                        | Best longer-term card architecture                    |

## What to change first

### 1. Introduce an explicit layout epoch and update generation

Use two separate counters:

* A **layout epoch** changes when the card structure or material geometry changes.
* A **HASS update generation** changes for every fan-out of a new `hass` value.

Every delayed observer or animation-frame callback must verify both values before mutating locks. This prevents an old telemetry update from clearing locks belonging to a newer update or a new layout.

Increment the layout epoch and synchronously clear all floors on:

* Disconnect and reconnect.
* Material width or column-count change, including the desktop/mobile transition.
* Card reorder and reset order.
* Hide and unhide.
* Entering and leaving show-hidden/customization preview.
* Most importantly, the transition from customization mode active to inactive, after preview-only hidden cards are removed.
* Replacement of the rendered card list.

Do not reset merely because host height changed during a normal state tick.

### 2. Track stable heights per layout item

Use one shared `ResizeObserver` to maintain the last stable border-box height for each rendered, non-hidden layout item. Here, “visible” should mean participating in the dashboard layout—not merely intersecting the viewport. Cards above the viewport are exactly the ones that need protection.

Prefer the cached observer measurements in the hot `hass` path. This avoids forcing a synchronous layout read for every card every two seconds. When a cache is unavailable after a structural reset, do one batched read to establish the baseline.

Use a wrapper-level CSS variable and class rather than overwriting arbitrary inline card styling:

* Apply `min-block-size`, not fixed `height`, so cards can still grow.
* Ensure the wrapper uses `box-sizing: border-box` if the cached value is a border-box measurement.
* Round upward to the nearest CSS pixel so fractional measurements cannot under-lock the item.
* Remove only the integration-owned property and class during cleanup.

### 3. Lock every item before forwarding `hass` to any card

The ordering is critical:

1. Collect the complete set of active layout items.
2. Resolve every baseline height.
3. Apply floors to **all** items in a single write phase.
4. Only then forward `hass` to the first card.
5. Forward the update to the remaining cards.
6. Schedule settlement and release.

Do not snapshot, lock, and update cards one at a time. The first card could collapse and change the geometry used to snapshot later cards.

This also means card-setter failures need cleanup protection. A `try/finally`-equivalent lifecycle should ensure that locks are eventually settled or cleared even when one card throws.

### 4. Use a temporary host floor, not a max-ever ratchet

For the first per-item candidate, keep a host-level floor equal to the **pre-update stable host height**, scoped to that HASS generation. It provides protection against layout-container or masonry bookkeeping that is not fully represented by item floors.

Do not continuously replace it with the maximum height ever observed across the entire layout epoch. The intended sequence is:

* Apply the current stable host height before card updates.
* Keep it while item locks are active.
* Release it only after the item floors have settled and been removed.
* Confirm that the natural final host height has returned to the expected value.

If the reporter’s iOS validation proves that per-item floors alone are sufficient, remove the host floor. If a temporary host floor is still required, keep it update-scoped. Preserve the monotonic epoch-wide ratchet only as a last fallback, and only with comprehensive reset and legitimate-shrink coverage.

### 5. Release locks based on natural card settlement

Do not use the locked wrapper’s size as the sole settlement signal. Its `min-block-size` can mask the natural child shrink, causing the observer to report the imposed floor rather than the real card size.

Observe or measure the inner card host/content node instead. Release after:

* At least one rendering opportunity following the `hass` fan-out.
* No natural-size change for two consecutive animation frames.
* The update generation and layout epoch still match.
* The item is still connected and belongs to the active card set.

A newer HASS generation should take ownership of the existing floor rather than allowing the older callback to remove it.

Avoid a blind timer such as “remove after 100 ms.” Chart initialization, device load, background throttling, and custom elements can make that intermittently too early.

For a genuine final shrink, the floor should eventually release; otherwise the per-item mechanism becomes another permanent ratchet. The regression should distinguish:

* **Transient collapse followed by restoration:** no exposed shrink at all.
* **Legitimate stable shrink:** one settled layout change, no retained stale floor.
* **Stable shrink near the bottom:** verify whether the temporary host backstop is required to prevent a clamp during release.

If current cards perform a genuine stable shrink on every telemetry update, that is the point at which the affected card needs a static shell—not a reason to retain permanent blank height globally.

## Exact failure modes to inspect

### Host-level ratchet

1. **Customization exit:** previewed hidden cards increase the maximum; leaving customization removes them but does not lower the ratchet.
2. **Transient oversize capture:** loading/error/chart initialization briefly makes the layout taller, permanently creating blank space.
3. **Breakpoint transition:** a one-column mobile height is retained when returning to three columns, or vice versa.
4. **Local offset movement:** one card above the viewport shrinks while the host’s empty space remains at the bottom; the total height is stable but lower cards move upward.
5. **Self-masking observation:** once `min-height` controls the observed host height, the host observer can no longer discover the smaller intrinsic height.
6. **Hidden-state observations:** `ResizeObserver` can notify on insertion, removal, and `display:none`; zero or hidden measurements must not become a new baseline. ([W3C][2])
7. **Reconnect leakage:** observers or inline floors survive disconnection and apply to newly built cards.

### Per-item transient floors

1. A card receives `hass` before every item is locked.
2. A stale animation-frame callback unlocks the next update’s floors.
3. The locked wrapper is used to infer natural settlement.
4. Existing inline `min-height` values are overwritten rather than restored.
5. Hidden or customization-preview items contribute zero or obsolete baselines.
6. Back-to-back updates inherit an old oversized floor indefinitely.
7. Applying the floor triggers the masonry relayout path on every tick.
8. A card grows during rendering but is clipped because fixed `height` was used instead of `min-height`.
9. Border/padding differences make the applied content-box minimum smaller than the measured border-box height.
10. Item removal while locked leaves observer registrations or style state behind.

### Atomic/static-shell refactor

An off-DOM swap is not automatically safe. Charts and custom elements often need connected layout dimensions, so a detached tree may initialize with zero width or height. The swap can also expose a smaller shell while nested elements upgrade asynchronously.

A static shell with incremental text, attribute, and chart-series updates is stronger. The existing energy-flow card demonstrates the desired rendering model. However, refactoring ForecastSummary, BatteryHealth, and the 24-hour chart now would broaden the release substantially, and it would still not protect future cards that replace their subtrees. The layout boundary therefore remains the correct generic containment layer for this bug.

## Regression coverage to add

The current 37 passing tests are a useful base, but the release regression should explicitly exercise geometry and lifecycle behavior.

### Core update lifecycle

* Assert that every active item floor is installed before the first child `hass` setter runs.
* Simulate a child changing from its normal height to zero or a small height, then returning to normal.
* Assert that host scroll range never falls below the pre-update range.

* Assert that all floors are removed after stable restoration.

### Generation races

* Start update A, queue its unlock, then start update B before A settles.
* Run A’s delayed callbacks.
* Assert that B’s floors remain intact.
* Settle B and verify complete cleanup.

### Real shrink and growth

* Final height larger than baseline: growth is visible and the floor releases.
* Final height smaller than baseline: floor eventually releases rather than becoming a permanent per-item ratchet.
* Transient oversize followed by normal height: no maximum is retained.
* Fractional heights: applied floor is never smaller than the measured size.

### Layout modes

* One-column mobile transient collapse.
* Three-column desktop collapse in one column without repositioning unrelated columns.
* Width change that does not alter material geometry: no full reset or masonry relayout.
* Breakpoint/column-count change: old floors and cached heights are discarded before rebaseline.

### Customization and structure

* Enter show-hidden/customization mode.
* Preview hidden cards and allow them to affect height.
* Leave customization mode and remove previews.
* Assert that the layout epoch changes, all old floors are cleared, and the new stable height is accepted.
* Exercise hide, unhide, reorder, and reset-order independently.

### Connection lifecycle

* Disconnect while an update is unsettled.
* Assert observers, queued frame callbacks, cached heights, classes, and CSS variables are cleared.
* Reconnect and verify the first new baseline does not inherit the old maximum.

### Performance invariant

Instrument the masonry/layout entry point and assert that ordinary telemetry updates:

* Do not reorder cards.
* Do not recompute column placement.
* Do not invoke a height-only full layout.
* Use cached observer heights or at most one batched measurement, never alternating read/write work per card.

The most important runtime assertion is a three-value invariant around each simulated telemetry update:

* `scrollTop` remains unchanged while the user is idle.
* `scrollHeight` does not transiently fall.
* The selected lower-card anchor’s top relative to the viewport remains unchanged.

Source-string checks can remain as smoke coverage, but they should not be considered the regression proof.

# PowerSync periodic mid-slot force review — 2026-07-15

## Pro response 1: approach decision

**Decision: ADJUST**

The direction is correct, but the invariant must be enforced in execution rather than by changing the optimizer schedule. A cached boundary action that was actually applied must not be reversed inside the same slot by a periodic solve unless a higher-priority trigger overrides it.

Required constraints:

- Apply only to the polling-loop solve.
- Track the boundary action that was actually executed, with wall-clock `slot_start` and `slot_end`.
- Gate only a non-forced boundary action changing to `charge`, `discharge`, or `export` in that same slot.
- Do not block safety/demand handling, user or settings actions, price-triggered re-optimization, startup, or continuation of an already-forced state.
- Do not mutate the published schedule; keep deferral as execution metadata.
- Use timestamps rather than a schedule index so replacement of the schedule during the solve cannot misidentify the committed slot.
- Let the commitment expire naturally at `slot_end`; the next cached boundary execution then owns the next slot.
- Clip force duration to the actual wall-clock time remaining in the contiguous action window. This is useful hardening but is not the root fix.

The smallest robust shape is to record boundary metadata when cached execution succeeds and pass a per-run trigger into the execution path. Before a polling solve issues a newly forced action, compare it with the unexpired non-forced boundary metadata and defer the hardware command if it would reverse the boundary decision.

## Pro response 2: fastest compliant implementation path

1. Add transient coordinator state for the last accepted boundary execution:
   `slot_start`, `slot_end`, the actual action, and whether it was forced. Do not persist it.
2. Extend `_run_optimization` with an explicit per-call execution trigger. The polling loop passes `poll`; price, startup, settings, and manual paths remain override-capable. Pass the trigger directly to `_execute_optimizer_action` so there is no sticky cross-run state.
3. In `_execute_cached_current_action_if_changed`, record metadata only after the cached action has been retained or successfully applied. Derive bounds from the action timestamp plus the configured interval.
4. In `_execute_optimizer_action`, place the guard after monitoring, active-force ownership, calibration, demand, export-price, EV-preserve, idle-exit, and reserve safety logic have produced the effective action, but immediately before any new charge/discharge/export hardware command.
5. If the trigger is `poll`, the boundary record is unexpired and non-forced, and the effective action is newly forced, log the deferral and return without changing execution markers. The fresh schedule remains published, so its force action starts at the next cached boundary.
6. For `allow_boundary_overrun=False`, compute the contiguous block end and use the ceiling of its remaining wall-clock minutes when now is inside the block. Preserve the existing fallback for out-of-window test fixtures or missing timestamps.
7. Regression coverage must include:
   - cached `self_consumption` at 13:10, a slow periodic solve wanting charge/export at 13:13, no force at 13:13, then force at the 13:15 boundary;
   - price-triggered execution bypassing the deferral;
   - a forced boundary action not being weakened;
   - no boundary metadata at startup;
   - expiry at the next slot;
   - 13:13:16 in a block ending 13:20 returning 7 minutes, not 10.
8. Verify with `python3.12 -m pytest` on the new regression tests first, then `tests/test_disable_and_double_command.py`, `tests/test_battery_export_allowed_slots.py`, and adjacent optimizer tests.

Critical traps called out by the review:

- guarding before safety logic;
- changing the schedule instead of execution;
- identifying the slot by list index;
- recording a merely planned action rather than an applied/retained one;
- blocking price or user/settings triggers;
- leaving trigger state sticky across runs;
- clipping by full slot count rather than absolute remaining time.
